This page gets InfiniteCode running in an existing repository with a
model provider connected.

If this is your first time with InfiniteCode, use the **DeepSeek
official API** through InfiniteCode's Anthropic-compatible provider
path. That path is the simplest default when you want provider-hosted
tools such as `web_search` and `web_fetch`.

## 1. Install InfiniteCode

Linux or macOS:

```bash
curl -fsSL https://raw.githubusercontent.com/AFS-Agentics/InfiniteCode/main/install.sh | sh
```

Windows PowerShell:

```powershell
irm 'https://raw.githubusercontent.com/AFS-Agentics/InfiniteCode/main/install.ps1' | iex
```

Verify the install:

```bash
infinitecode --version
```

You should see the version reported (for example, `infinitecode 0.1.31`).

## 2. Get a provider API key

Pick whichever provider you want. DeepSeek is a good first choice
because its Anthropic-compatible endpoint is well documented and
works on day one. Create an API key in your provider's dashboard.

The current provider that ships in onboarding defaults is **DeepSeek
Anthropic-compatible**:

| Field | Value |
| --- | --- |
| Invocation method | `Anthropic Messages` |
| Base URL | `https://api.deepseek.com/anthropic` |
| Model name | `deepseek-v4-pro` (or `deepseek-v4-flash` for lower cost) |

For OpenAI-compatible paths and self-hosted endpoints, see
`infinitecode.toml` in [Settings reference](/docs/configuration/settings).

## 3. Onboard

```bash
cd /path/to/your/repo
infinitecode onboard
```

If you already have an `infinitecode.toml`, onboarding will skip the
provider step. Otherwise, fill in the prompts.

## 4. Start InfiniteCode

```bash
infinitecode
```

If onboarding configured a provider, the TUI opens straight to the
composer. If not, onboarding re-prompts for the missing fields.

## 5. Ask a read-only first prompt

```text
Explain this repository's main runtime path. Do not edit files yet.
```

If the answer looks wrong, fix the setup before asking for changes.
The most common causes are wrong working directory, wrong model name,
wrong base URL, or a missing API key.

## 6. Run a scoped change

```text
Update the install docs to mention offline installation.
Keep the change scoped to docs only.
```

When you already know the file, attach it with `@`:

```text
Update @apps/docs/content/docs/quickstart.mdx to add a v0.1.31 callout.
```

When you know the exact shell command, use `!` shell mode:

```text
!git status --short
```

## 7. Try the v0.1.31 capabilities

If you want to test the Self-Review pipeline, run any scoped change
with `--self-verify`:

```bash
infinitecode --self-verify "Refactor the user record updater"
```

If you're using the **Desktop** app, try voice input from the
composer rail — see
[Desktop voice input](/docs/features/desktop-voice) for the trial
flow.

For web search, ask any freshness-sensitive question:

```text
What's the latest stable version of bun as of this week?
```

The answer should arrive with inline citation chips.

## 8. Verify the change

```text
Run the checks for this docs change and tell me what passed or failed.
```

Then inspect the working tree:

```text
!git status --short
```

Review the diff before committing or continuing. InfiniteCode can
help explain the diff, but the repository state is still yours to
accept.
