---
title: First Task
description: Run a practical first coding task with InfiniteCode.
---

Use the first task to prove three things:

1. InfiniteCode is running in the repository you intended.
2. The model can read and reason about the project.
3. InfiniteCode can make a small scoped change and run the right checks.

Start with read-only context. Move to edits only after the answer matches the
repository.

## Start In A Repository

Open a repository before launching InfiniteCode:

```bash
cd /path/to/your/repo
infinitecode
```

InfiniteCode uses the current working directory as the session workspace. If you launch
it from the wrong directory, file search, context, and validation commands will
all point at the wrong project.

## Ask For The Runtime Path

Use a read-only prompt:

```text
Explain this repository's main runtime path. Do not edit files yet.
```

For a web app, ask for request flow:

```text
Explain how a docs page is loaded and rendered. Include the files I should read first. Do not edit files yet.
```

For a Rust crate, ask for crate boundaries:

```text
Explain the crates involved in provider configuration. Do not edit files yet.
```

Read the answer before continuing. If the answer names unrelated files or misses
obvious entry points, correct the working directory or model setup first.

## Point At Known Files

When you already know the file, attach it with `@`:

```text
Review @apps/web/content/docs/get-started/install.mdx for missing install paths. Do not edit yet.
```

Use `@` when you want InfiniteCode to inspect specific context instead of searching
from scratch.

## Ask For One Scoped Change

Keep the first edit small:

```text
Update @apps/web/content/docs/get-started/install.mdx so it covers offline installation. Keep the change scoped to this page.
```

Good first tasks have a narrow file set, a clear desired behavior, and an
obvious verification command.

Avoid broad first prompts such as:

```text
Improve the docs.
```

That gives InfiniteCode too much freedom before you have verified the setup.

## Run Validation

Ask InfiniteCode to run the relevant checks:

```text
Run the docs lint and build checks for this change.
```

If you know the exact command, use `!`:

```text
!npm run lint
```

For Rust changes, prefer package-scoped checks first:

```text
!cargo test -p infinitecode-tui status_indicator -- --nocapture
```

Then broaden only when the change touches shared behavior.

## Review The Diff

Inspect the working tree:

```text
!git status --short
```

Ask InfiniteCode to summarize the diff:

```text
Summarize the docs changes and call out anything unrelated.
```

If unrelated files changed, do not stage them until you understand why they are
dirty. InfiniteCode should work with the existing worktree and avoid reverting unrelated
changes.

## Continue Or Stop

After the first task:

- Continue with another small scoped change if the setup behaved correctly.
- Fix provider, model, working-directory, or permission issues before expanding
  scope.
- Use [@ Fuzzy File Search](/docs/using-infinitecode/fuzzy-file-search) for known files.
- Use [! Shell Mode](/docs/using-infinitecode/shell-mode) for exact commands.
- Use [Architecture Overview](/docs/contributors/architecture-overview) before
  changing runtime code.
